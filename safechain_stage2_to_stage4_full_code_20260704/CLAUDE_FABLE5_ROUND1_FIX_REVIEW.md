## 3. New Findings Introduced by Round 1

Severity scale as before: blocker / high / medium / low / info. **There are no new blockers or highs.**

- **R1-1 (medium, perf/memory — fix before GPU launch).** `compute_loss` now builds the diagnostics dict on **every microstep**, not just on log steps (`pause_kl_trainer.py:457-462`), and `_pause_logit_diagnostics` materializes `shift_logits[non_pause_mask].float()` plus a full-vocab `F.softmax` over **all labeled non-pause positions** (`:228-233`) — the exact N×V float32 materialization the chunked suppression loss (`:383-397`) was written to avoid, and without `torch.no_grad()` (the softmax is autograd-tracked until scope exit; it never reaches backward, so gradients are untouched, but the graph holds the tensor alive). Worst case at `max_seq_length: 4096`, bs 2 (1.5B, V=151,936): ~5 GB for the indexed copy + ~5 GB for the softmax ≈ **10 GB transient per microstep** on a 48 GB A6000, on top of student+teacher logits; 8B (V=128,256) is proportionally similar. This is fail-stop (OOM), never silent corruption, and with realistic completion lengths it will usually fit — but it is a needless mid-run-crash risk and a throughput tax (full-vocab softmax + `.cpu()` syncs 800×/run for numbers logged 16×). Cheap fix: compute diagnostics only when `_maybe_log_loss_parts` will actually log, wrap in `torch.no_grad()`, and reuse the suppression chunk pattern for the non-pause probabilities (argmax needs no fp32 copy at all).
- **R1-2 (medium, silent fallback).** `find_matched_row` falls back to **prompt-key matching** when ID lookup misses (`extract_hidden_states.py:181-192`). On the wired path this is dead code (prep writes shared IDs; shards are ID-joined), but on any manually assembled or stale input, a cotpause row silently gets a control from a *different trajectory for the same prompt* — and this corpus has multiple trajectories per prompt with different labels (that is what `split_source_label_prompt_group` exists for). No label-equality check is performed on the matched row. Recommend: drop the prompt fallback (drop the row instead), or at minimum count `matched_by_prompt` separately in the manifest and assert `trajectory_safety_label` equality. Related low: for rows without explicit `id`, `row_id()`’s hash includes the pause-bearing output (`:161-166`), so cotpause/nopause hash IDs can never match and everything would go through the prompt fallback — again only reachable with hand-made files.
- **R1-3 (low, doc/UX — verified empirically).** Setting `STAGE4_8B_FULL_SFT_CONTROL` does **not** unblock the 8B liveness plan: the static `positive_control_status: missing_required_full_sft_pause_control` in `stage4_pause_gprs_8b_4xa100.yaml` keeps `liveness_plan_status` at `blocked_missing_positive_control` (confirmed by execution with the env var set). Fail-safe in the right direction — a human must also flip the status after verifying the control is genuinely full-SFT-class — but nothing documents that two edits are required, and the fix-request wording ("Requires STAGE4_8B_FULL_SFT_CONTROL and otherwise marks liveness as blocked") implies the env var suffices. Add a comment in the config.
- **R1-4 (low, host memory).** The invariant callback now retains full embedding snapshots on CPU for the **entire run** (`snapshots` no longer freed after the first check): ~0.45 GB/rank for tied 1.5B, ~2.1 GB/rank (×4 ranks ≈ 8.4 GB host RAM) for untied 8B. Fine on the target nodes; worth knowing it exists.
- **R1-5 (low).** `pipeline.py:238` bakes `--allow_learned_delta` into the generated archival plan command. Plans are not auto-executed and the note says "Archival learned-delta reproduction only", but the acknowledgement flag is now one copy-paste away from any reader of the plan JSON. Consider leaving the flag out and letting the runner refuse instructively.
- **R1-6 (low).** Matched controls are hardcoded to offsets `[3,4]` (`extract_hidden_states.py:712`), matching the insertion offsets and the pooled `control_cot3_cot4` spec for both 1.5B (cot3) and 8B (cot4) — but the configured `cot_offsets: [3,4,7,8]` means `cot_7/cot_8` screens have no matched controls. Acceptable now; parameterize if deeper-offset comparisons become claims.
- **R1-7 (info, migration).** `stable_probe_id` changes the fallback-ID scheme (hash of source‖prompt‖reasoning‖final instead of prompt+output), so re-prep produces different IDs than old data dirs. Because extraction now *requires* `nopause/{split}.json` (SystemExit if missing, `extract_hidden_states.py:598-603`; `run_intra_pause_probe_full.py:419` always passes the flag), old prepared dirs fail fast rather than mixing generations. Correctly fail-closed; just plan the re-prep.
- **R1-8 (low, pre-existing trust).** If a raw source supplies duplicate `id` values across genuinely different rows, `no_pause_by_id` (last-wins) / `build_matched_lookup` (first-wins) can pair a row with its duplicate’s control. Same-fingerprint duplicates are harmless (identical content); distinct rows sharing an upstream ID would be silently cross-matched. Consider a duplicate-ID assertion in prep.
- **R1-9 (info, perf).** The control forward runs while the main batch’s all-layer `hidden_states` tuple is still alive (`extract_hidden_states.py:800-836`), roughly doubling peak extraction memory and total forward cost. If extraction OOMs, `del hidden_states` before the control forward or halve `--batch_size`. Also, `--limit` truncation applies only to cotpause rows — harmless, since controls are looked up by ID.
- **R1-10 (info, DDP — checked, no issue).** The new diagnostics and invariant checks introduce **no collectives**, consume no RNG, and never touch the loss graph, so gradients, data order, and DDP synchronization are bit-identical to the pre-diff trainer. Invariant checks run per-rank on replicated weights, so a violation raises on all ranks together (fail-stop, no hang in practice under `torch.distributed.run`). `_embedding_weights` uses `get_input_embeddings()` with fixed names and `_unwrap_model`, so DDP wrapping cannot desynchronize the snapshot keys.

## 4. Direct Answers to the Six Review Questions

1. **Is Stage2 1.5B still GO?** Yes. The loss path is byte-identical in effect (all new code is detached diagnostics); rows-only enforcement is strictly stronger (periodic + train-end bit-compare, tested); the launcher now hard-asserts the pause token. The one caveat is R1-1: gate the diagnostics to log steps before launch to remove an avoidable OOM/throughput risk. This is launch hygiene, not correctness.
2. **Did the diagnostic/invariant changes introduce training-objective or DDP risk?** No objective risk (nothing new contributes to `loss`; verified every new tensor ends in a detached scalar) and no DDP risk (no collectives, no RNG, per-rank checks on replicated weights — R1-10). The real costs are transient memory and step-time overhead (R1-1) plus persistent CPU snapshots (R1-4).
3. **Are `control_cot_3/4` now true no-pause matched controls?** Yes. Same raw trajectory, shared stable ID, separate pause-free forward pass, positions independently located in no-pause coordinates that address the same content tokens as the legacy pause-excluded coordinates; failures drop rows with named counters instead of aliasing. Residual weaknesses: the silent prompt-key fallback and missing label-equality check (R1-2), controls limited to offsets 3/4 (R1-6), and unavoidable minor tokenization-boundary drift at the insertion site (inherent to the design, not a bug).
4. **Is the teacher-forced Stage3 screen honestly labeled?** Yes. `primary_endpoint: teacher_forced_pause_minus_prompt_screen` says exactly what it is; `within_prompt_auroc` is demoted to a `confirmatory_endpoint` with `status: not_implemented` and its prerequisite named; the on_policy stubs in both KL configs are marked `not_implemented`. No config now claims an unimplemented endpoint as current.
5. **Is Stage4 sufficiently fail-closed?** Yes, and I verified it by running the gates: learned-delta refuses every non-validate phase without an explicit double-opt-in (flag or config key; shell requires `ALLOW_LEARNED_DELTA=true`); GPRS eval/generation/judge/summary/all are blocked even under `--dry_run`; the bare default invocation resolves to GPRS validate-only; the liveness runner still hard-refuses non-dry execution and now writes `blocked_missing_positive_control` for the 8B config; `TARGET_SPECS` can no longer be injected via environment. Remaining soft spots are R1-3 (status-string interplay undocumented) and R1-5 (ack flag embedded in the archival plan) — both cosmetic relative to the gate itself. The `acknowledge_deprecated: true` config bypass is an accepted second key, default-false, and grep confirms no config or shell sets it.
6. **Any blocker/high issues before landing into the main repo?** No. Recommended before/with landing, in order: (1) R1-1 — gate diagnostics to log steps + `no_grad` + chunked softmax (protects the imminent 1.5B launch); (2) R1-2 — remove or instrument the prompt fallback and assert control-label equality; (3) R1-3 — one comment line in the 8B config. None of these invalidates the diff as written.
